# Localization (i18n)

`contributes.i18n` declares the language-pack file for each locale; every `%key%` placeholder in
the manifest is resolved through it at display time.

```jsonc
"i18n": {
  "en": "i18n/en.json",
  "zh-CN": "i18n/zh-CN.json"
}
```

- Keys = locales (currently `en` and `zh-CN` are supported), values = language-pack JSON file
  paths (relative to the plugin files root).
- Only declared locales are loaded.

## Language-pack file format

Flat JSON: key → text. The key is the manifest `%key%` name with the percent signs removed.

`i18n/zh-CN.json`:

```json
{
  "cmd.sayHello": "打个招呼",
  "set.greeting": "问候用语",
  "set.greeting.desc": "打招呼时使用的词",
  "set.loud": "大声问候",
  "panel.hello": "问候面板"
}
```

`i18n/en.json`:

```json
{
  "cmd.sayHello": "Say Hello",
  "set.greeting": "Greeting word",
  "set.greeting.desc": "Word used when greeting",
  "set.loud": "Shout the greeting",
  "panel.hello": "Greeting Panel"
}
```

## Where `%key%` is used

Display text in commands, settings and panels supports `%key%` placeholders:

- `command.title`
- `setting.title` / `setting.description`
- `panel.title`

For example `"title": "%cmd.sayHello%"` is replaced at display time with the `cmd.sayHello` text
in the current language.

## Runtime behavior

- Language packs are mounted into the frontend **overlay** at runtime under the key space
  `plugin.<id>.*` (plugins never collide).
- **Lookup chain**: current language → `en` → the key name verbatim. That is, when a key lacks a
  translation in the current language it falls back to `en`, and if still missing it displays the
  key name itself (handy for spotting missing translations during development).
- The app reads language-pack JSON via `plugin_i18n_read`.

## Recommendations

- Keep `en` complete — it is the fallback language, so every key should at least have English.
- Dot-namespaced keys (`cmd.*` / `set.*` / `panel.*`) help organization, but this is only a
  convention; the key space is flat.
